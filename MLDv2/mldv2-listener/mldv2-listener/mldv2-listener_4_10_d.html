Test Label

MLD.4.10: Listening State Change under MLDv2 Query Reception
Part D: General Query and Listening Stop

Purpose

To verify that the MLDv2 Listener replies an optimized report, when MLDv2 Query message reception and state change occur continuously.

References

RFC3810 6.2. Action on Reception of a Query
RFC3810 6.3. Action on Timer Expiration

Discussion

None

Test Requirement

None

  • Common Test Setup
    1. Configure HUT to have:
      • Robustness Variable of 2
      • Query Interval of 125
      • Query Response Interval of 10 seconds
      • Unsolicited Report Interval of 1 second

  • Common Test Cleanup
    1. Return all timers and variables to their defaults values.
    2. Stop any listening. HUT does not listen to any multicast other than a solicited-node multicast address of an address assigned to the interface.

Procedure

Step 13. Enable MLDv2 on the HUT.
Step 14. Call ListenerAPI (S1, I1, M1, INCLUDE, {S1}) on the HUT.
Step 15. Observe the packets on all networks.
Step 16. TR1 transmits MLDv2 General Query message A.
    -----
    IPv6 Header
        IPv6 Hop Limit                  = 1
        IPv6 Source Address             = TR1 Link-local Address
        IPv6 Destination Address        = All Nodes Address
    Hop-by-Hop Options Header
        Router Alert Option
    ICMPv6 Header (MLD Query)
        Maximum Response Code           = 10000
        Multicast Address               = ::
        S Flag                          = 0
        QRV                             = 2
        QQIC                            = 125
        Number of Sources               = 0
    -----
Step 17. Call ListenerAPI (S1, I1, M1, INCLUDE, {NULL}) on the HUT.
Step 18. Observe the packets on all networks.

Observable Results

Step 15: The HUT must transmit 2 MLDv2 Report messages with Multicast Address Record ALLOW (S1) for M1.
Step 18: The HUT must transmit 2 MLDv2 Report messages with Multicast Address Record BLOCK (S1) for M1.
         And the HUT must transmit MLDv2 Report message with Multicast Address Records IS_EX (NULL) for HUT solicited-node multicast within 10 seconds from Step 16.

Possible Problems

Since HUT decides on time to transmit each MLDv2 Report message at random, the expected result may not be obtained. The expected result by several trials will be obtained. Besides, probability may be raised by adjusting Maximum Response Code of MLDv2 Query message.
